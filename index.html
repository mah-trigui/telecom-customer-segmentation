Telecom Customer Segmentation

From observation-unit design through unsupervised clustering to supervised rule-based assignment — a full CVM segmentation pipeline

View Repository Explore Method

Overview

This project delivered a full customer segmentation system for telecom CVM operations. It had two phases: unsupervised clustering to discover behavioral segments, then supervised classification to assign new customers to those segments using interpretable business rules.

The main challenge was not the clustering algorithm. It was the series of design decisions that came before and after: defining what one row should represent, making variables compatible with distance-based methods, iterating until clusters became operationally meaningful, and bridging unsupervised output to a production-ready supervised assignment system.

Target: 834K prepaid subscribers · Features: 174 behavioral variables · Tools: SAS, Oracle/Toad, R · Outcome: Operational CVM segments with automated new-customer assignment

1. Observation Unit Design

The hardest segmentation decision was not the clustering method. It was deciding what one row should represent.

With months of behavioral history per subscriber, the first design question was: what exactly should one row in the analytical base table represent?

Approach A — One Subscriber = One Row

Collapse the historical window into one behavioral profile: average ARPU, average recharge, average outgoing usage, average data activity. You segment customers.

Approach B — One Subscriber-Month = One Row

Keep each monthly snapshot as a separate observation. Customer_A in June ≠ Customer_A in July ≠ Customer_A in August. You segment behavioral states over time.

That choice changes the segmentation logic completely. A person-level segmentation gives stable long-term profiles useful for structural campaigns. A month-level segmentation reveals transitions: from low usage to recharge-driven activation, from voice-heavy to data-heavy behavior.

Code: Aggregated subscriber profile (6-month average)

/* One subscriber = one row: 6-month behavioral average */
SELECT DISTINCT subscrp_id,
       SUM(MNT_RECHARGE) / 6         AS MNT_RECHARGE,
       SUM(NB_RECHARGE) / 6          AS NB_RECHARGE,
       SUM(NB_JR_RECHARGE) / 6       AS NB_JR_RECHARGE,
       SUM(NB_RECHARGE_WEEKEND) / 6  AS NB_RECHARGE_WEEKEND,
       CASE WHEN SUM(NB_RECHARGE) > 0
            THEN SUM(MNT_RECHARGE) / SUM(NB_RECHARGE)
            ELSE 0 END               AS Rech_Moy,
       CASE WHEN SUM(NB_RECHARGE) > 0
            THEN SUM(NB_JR_RECHARGE) / 181
            ELSE 0 END               AS Freq_Rech
FROM trafic_recharge
GROUP BY subscrp_id;

Key insight: Does the business need stable customer types, or changing customer states? That question must be answered before any clustering method is chosen.

2. Iterative Profiling Loop

A cluster is useless until an iterative loop of profiling and redesign turns it into an operational segment.

Useful clusters did not emerge from running one method once. The final segmentation came from repeated cycles of representation, clustering, profiling, and redesign.

The business cannot act on Cluster 1, Cluster 2, Cluster 3. Operations needs: young prepaid users, data addicts, high consumers, old loyal subscribers, multisim households, fragile users.

The iteration loop

Represent → Cluster → Profile → Rethink → Rebuild

Profiling dimensions per iteration

  • Recharge behavior — frequency, amount, patterns, weekend vs weekday
  • Voice vs data mix — usage intensity by service type
  • Handset quality — 2G / 3G / 4G device tier
  • Network usage — on-net vs off-net, community structure
  • Region and mobility — geographic footprint, cell tower count
  • CVM response — campaign sensitivity, offer receptivity
  • Churn signals — fragility indicators, inactivity days

Code: Profiling by segment

/* Profile each segment across behavioral dimensions */
PROC MEANS DATA=abt_segmented N MEAN STD MEDIAN P99 MIN MAX;
  WHERE segment = 6;  /* e.g., Data Addicts */
  VAR MNT_RECHARGE revenu_total DUREE_APPEL_TOTAL
      MNT_FORFAIT_DATA VOLUME_FORFAIT NB_CELL
      NB_REGION deg_in NB_Comm Freq_Rech;
RUN;

/* Profiling by handset type across segments */
SELECT handset,
       COUNT(CASE WHEN segment=1 THEN subscrp_id END) AS SEG_1,
       COUNT(CASE WHEN segment=2 THEN subscrp_id END) AS SEG_2,
       COUNT(CASE WHEN segment=3 THEN subscrp_id END) AS SEG_3
FROM abt_segmented
GROUP BY handset;

Key insight: Unlike supervised learning, there is no target variable to tell you whether the representation is right. Good segmentation is designed through iteration — the iteration is the method.

3. Variable Clusterability

Before clustering telecom customers, I had to make the variables clusterable.

The raw feature space had extreme distributions: recharge amounts with skewness of 5+, call durations with skewness of 8+, off-net activity with skewness of 40+, some variables with kurtosis above 1000.

Standard scaling alone would make the distance geometry unstable. So before reducing redundancy, a more basic question had to be answered: can these variables even live in the same clustering space?

Variable-specific transformations

LOG

For moderate skew (1-3). Applied to: ARPM, NB_Comm, NB_CELL, nb_sms_total, VOIP

SQRT

For light skew. Applied to: NB_JOUR_ACTIVITE, PRCT_4G, P_DUR_OOR, P_PARTAGE

Power ^0.25

For heavy skew (5+). Applied to: MNT_RECHARGE, revenu_total, DUREE_APPEL_OOREDOO, VOL_FMS

EXP

For reverse / compressed distributions. Applied to: MULTISIM, P_FF_Data, PRCT_3G

Code: SAS Enterprise Miner transformation output

/* Actual transformations computed by the pipeline */
REP_IMP_revenu_total:       PWR  → (max(val - 0) / 38.93) ** 0.25
REP_IMP_ARPM:               LOG  → log(max(val - 0) / 0.7076 + 1)
REP_IMP_MNT_RECHARGE:       PWR  → (max(val - 0) / 51.5) ** 0.25
REP_IMP_NB_CELL:            LOG  → log(max(val - 0) / 293 + 1)
REP_IMP_NB_Comm:            LOG  → log(max(val - 0) / 57 + 1)
REP_IMP_PRCT_4G:            SQRT → sqrt(max(val - 0) / 0.977)
REP_IMP_VOIP:               LOG  → log(max(val - 0) / 136.11 + 1)
REP_IMP_VOL_FMS:            PWR  → (max(val - 0) / 39326.7) ** 0.25
REP_IMP_MULTISIM:           EXP  → exp(max(val - 0))

/* Result: 174 candidates → 49 INPUT, 49 REJECTED */
/* Rejected not for correlation — but because no transform
   could make them behave well enough for distance methods */

Key insight: Skewness killed more variables than correlation did. In distance-based unsupervised learning, bad variable shape is often a bigger problem than redundancy.

4. K-Means as Compression Layer

K-means can be a compression layer before hierarchical clustering.

With 834K subscribers, hierarchical clustering (O(n²)) cannot run directly. K-means scales but flattens nested structure if used alone.

The solution: use them in sequence. K-means is not the segmentation — it is the compression step.

Step 1: K-Means Compression

Compress 834K subscribers into 1000 representative centroids. Fast, scalable, captures density.

Step 2: Hierarchical Clustering

Run Ward's method on 1000 centroids. Finds nested structure and natural segment boundaries.

Code: Two-stage clustering

/* Stage 1: K-means compression */
PROC FASTCLUS DATA=normalized_abt
     MAXCLUSTERS=1000
     OUT=compressed_centroids;
  VAR transformed_var1 - transformed_var49;
RUN;

/* Stage 2: Hierarchical clustering on centroids */
PROC CLUSTER DATA=compressed_centroids
     METHOD=WARD
     OUTTREE=tree_output;
  VAR transformed_var1 - transformed_var49;
RUN;

/* Final segment assignment */
PROC TREE DATA=tree_output
     NCLUSTERS=8
     OUT=final_segments;
RUN;

Key insight: The first algorithm is not the segmenter — it is the reducer. In large unsupervised systems, one algorithm can serve as a modeling layer for another.

5. Supervised Rule-Based Assignment

The decision tree was not the model. It was the rule-extraction layer between unsupervised clusters and business operations.

The unsupervised phase gave meaningful clusters — but clusters alone cannot assign new customers. A subscriber who activated last month has only 1-2 months of history. The unsupervised model was built on 6 months. You cannot cluster them the same way.

So the supervised phase had a different goal: extract interpretable assignment rules from existing clusters, adjust thresholds with business logic, and apply them to new customers.

The bridge: from clusters to rules

1. Label

Assign unsupervised segment labels to existing customers as a target variable.

2. Train

Fit decision tree, neural network, logistic regression on the labeled data. Compare performance.

3. Extract Rules

From the decision tree: extract splitting thresholds and variable conditions.

4. Adjust

Round thresholds for business interpretability. Validate with CVM teams.

Why thresholds need business adjustment

/* Raw tree output */
IF ARPU_DATA > 7.34 AND NB_FORFAIT_DATA > 2.17 → "Data Addict"

/* Business-adjusted rule */
IF ARPU_DATA > 7 DT AND NB_FORFAIT_DATA > 2   → "Data Addict"

/* Consequence: original cluster had ~100K members
   Rule-based assignment may capture 90K or 110K
   That is acceptable — the goal is stability and interpretability */

Code: Building supervised ABT for new customers

/* New customers: activated in last 1-4 months */
/* Build ABT with available history (shorter window) */
CREATE TABLE supervised_abt AS
SELECT DISTINCT
       b.SUBSCRP_ID,
       b.OFFER_ID,
       INTCK('DTMONTH', b.SUBSCRP_CREATION_DT, reference_date) AS Anciennete_mois,
       CASE WHEN type_4g = 'true' THEN '4G'
            WHEN type_3g = 'true' THEN '3G'
            ELSE '2G' END AS HANDSET
FROM SUBSCRP_D b
WHERE SUBSCRP_STATUS_CD IN ('A','S')
  AND SUBSCRP_TYPE_CD = '11'
  AND SUBSCRP_ACTIVATION_DT > cutoff_date;

Models compared

  • Decision Tree — primary choice for rule extraction and interpretability
  • Neural Network — higher accuracy but black-box, used as benchmark
  • Logistic Regression — baseline comparison
  • Gradient Boosting — ensemble benchmark

Key insight: Discovery is unsupervised. Assignment is supervised. The bridge between them is rule extraction with business-adjusted thresholds. The model that assigns is not the model that discovered.

Where This Pattern Generalizes

  • Banking customers with monthly transaction states
  • E-commerce users with lifecycle behavior changes
  • Subscription products with engagement-state segmentation
  • IoT / device populations with operating-state clustering
  • Any domain where unsupervised discovery must become production assignment

The reusable pattern: before clustering entities, decide whether you are clustering people, states, or transitions. Then bridge from discovery to operational rules.
